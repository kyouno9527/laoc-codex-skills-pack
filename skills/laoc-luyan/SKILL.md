# laoc-luyan

laoc-luyan 负责裁决：对照原始诉求和真实产物，用证据判断通过、部分通过或打回。

## 何时用

- laoc-luxing 已经产生改动、文档、数据、配置、服务或交付物。
- 鹏哥问“对不对”“验一下”“review 一下”“能交了吗”。
- 准备说完成、交付、上线、归档之前。
- 外部 agent、老G、模型或同事给了结果，需要抽验。

## 核心步骤

1. 对账：逐条对照鹏哥原始请求、显性要求、隐含成功标准和 laoc-luxing DoD。
2. 四层验收：
   - 存在：文件、接口、页面、数据、命令是否存在。
   - 实质：内容是否真覆盖需求，不是空壳、占位或伪实现。
   - 接线：系统是否实际引用、加载、调用、发布或交付。
   - 功能：真实运行或人工检查是否符合预期。
3. 实跑取证：测试、构建、lint、脚本、接口请求、页面检查、数据校验或人工复核。
4. 风险扫描：查 TODO、FIXME、placeholder、coming soon、硬编码假数据、空 handler、只打印日志的实现。
5. 边界复核：检查是否误写敏感资料、错误位置、错误入口、错误受众或未授权工作区。
6. 裁决：明确通过、部分通过或打回；失败时说明回 laoc-luxing 修，还是回 laoc-luwen 重诊。

## 输出格式

```markdown
裁决：通过 / 部分通过 / 打回

对账：
- 原要求：...
- 结果：...

证据：
- ...

缺口：
- ...

下一步：
- 进 laoc-lujie / 回 laoc-luxing hotfix / 回 laoc-luwen 重诊
```

Review 请求使用：

```markdown
发现：
- 严重级别 · 位置 · 问题 · 影响 · 建议

未覆盖：
- ...
```

## 禁止事项

- 禁止没跑验证就说完成或通过。
- 禁止把“文件存在”当“功能实现”。
- 禁止只信老G或外部汇报，不抽验关键产物。
- 禁止把延期项、豁免项伪装成完成。
- 禁止为了给出好消息而弱化风险。
