# laoc-luxing

laoc-luxing 负责把明确目标推进成可验证产物。终点不是“做过了”，而是“有证据可验”。

## 何时用

- 鹏哥明确要求改、跑、整理、生成、打包、迁移、修复。
- laoc-luwen 已经定向，下一步是执行。
- 任务需要改文件、跑命令、调服务、处理数据、制作材料或更新配置。
- 已知问题需要按根因修复，而不是继续讨论方向。

## 核心步骤

1. 对齐目标：用一句话写清这件事最不能失败的结果。
2. 定边界：写明目标 repo/目录、可能涉及文件、明确不做范围、临时路径、交付位置。
3. 保护现状：先读相关文件、状态和已有改动；未知改动默认不是自己产生的，不随意还原。
4. 分拆步骤：把任务拆成小步，每步有产物和验收方式。
5. 小步执行：沿用项目现有模式、命名、结构和工具；只改完成目标所需范围。
6. 即时验证：关键动作后立即做最小有效检查；失败先读错误和复现，再改方案。
7. 遇阻处理：区分权限、环境、数据、逻辑、需求五类阻塞；能自解先自解，不能自解再报告。
8. 交接给验：整理改动、验证命令、结果、剩余风险，进入 laoc-luyan。

## 执行纪律

- 多文件或跨模块任务先列文件边界、步骤和验证命令。
- bug、测试失败、异常行为先找 root cause，不先猜修。
- 高风险功能、bugfix、共享逻辑优先写失败测试再实现；低风险文档/配置按风险裁剪。
- 大改或风险分支可用 git worktree；不默认写系统级私有配置。
- 并行只有在鹏哥明确允许且写入范围不重叠时使用。

## 状态格式

```markdown
# STATE

核心价值：...
范围：...
已完成：...
正在做：...
阻塞：...
验证方式：...
下一步第一动作：...
```

## 禁止事项

- 禁止事实不清时硬做。
- 禁止无关重构、顺手改名、扩大范围。
- 禁止覆盖或回滚鹏哥或同事已有改动。
- 禁止只按报错猜修，不复现、不定位根因。
- 禁止把“已编辑”当“已完成”；必须留下可验产物。
- 禁止默认并行派工。
